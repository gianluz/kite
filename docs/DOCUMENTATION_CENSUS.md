## ğŸ¤” Key Decisions

### Question 1: Do we need API docs (Dokka)?

**Answer**: YES, but not urgently.

**Reasoning**:

- User-facing documentation is more important right now
- DSL reference (manual) is more useful than API docs for users
- API docs are helpful for:
    - Contributors understanding internal APIs
    - Users who want to extend Kite
    - Advanced users needing implementation details

**Recommendation**:

- Priority 4 - Generate after user docs are complete
- Focus on public APIs only (exclude internal packages)

### Question 2: Separation of User vs Developer docs?

**Answer**: YES, absolutely.

**Reasoning**:

- Users don't care about Kite's internal architecture
- Users want: "How do I use Kite?" not "How does Kite work?"
- Developers need: architecture, testing strategy, contribution guide
- Mixing them creates confusion

**Clear separation**:

- `docs/user/` - For Kite users (people using Kite in their projects)
- `docs/developer/` - For Kite developers (people contributing to Kite)
- `docs/examples/` - Working examples (useful for both)

### Question 3: What to do with specs/?

**Answer**: KEEP, but update to match reality.

**Reasoning**:

- Specs are design documents - useful for understanding decisions
- Some specs are outdated (builtin-features, platform-integration)
- Should reflect what was actually built, not the original plan

**Action**:

- Keep specs/ as historical design documents
- Update to match implementation
- Add notes where implementation diverged from spec
